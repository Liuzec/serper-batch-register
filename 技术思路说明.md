# Serper 注册机 — 技术思路说明

> 本文用非技术语言解释这个工具是怎么工作的，以及为什么这样设计。

---

## 一、我们要做什么

Serper.dev 是一个提供 Google 搜索 API 的网站。注册一个账号就能拿到一把"钥匙"（API Key），用这把钥匙可以让程序自动搜索 Google。

我们的目标：**批量自动注册 Serper 账号，收集这些钥匙**。

手动注册一个账号大概需要 2-3 分钟（填表单、收邮件、点验证链接、复制钥匙）。如果要注册 100 个，手动操作就不现实了。

---

## 二、正常人注册一个账号要做什么

假设你自己去注册，步骤是：

1. 打开 serper.dev/signup 页面
2. 填写姓名、邮箱、密码
3. 网页弹出一个"我不是机器人"验证（你可能感知不到，因为它在后台自动完成）
4. 点击"注册"按钮
5. 去邮箱里找一封验证邮件，点击里面的链接
6. 回到网站，登录
7. 在后台页面看到你的 API Key

关键难点在于：网站不希望有人批量注册，所以设置了多道防线。

---

## 三、网站的四道防线

### 防线 1：验证码 — "证明你是人类"

Serper 用了 **两套验证码系统**，同时生效：

- **Google reCAPTCHA v2**（隐形版）：Google 的验证系统，在你点击注册时悄悄在后台判断你是不是机器人
- **Cloudflare Turnstile**：Cloudflare 的验证系统，同样在后台运行

注册时必须同时提交这两个验证码的"通行证"，缺一不可。

### 防线 2：邮箱验证 — "证明你有真邮箱"

注册后网站会发一封邮件到你填的邮箱，你必须点击里面的链接才能激活账号。这意味着不能随便填一个假邮箱。

### 防线 2.5：一次性邮箱域名黑名单 — "别用假邮箱"

Serper 维护了一份一次性邮箱域名黑名单。如果注册时使用了黑名单上的邮箱域名，服务器会直接返回 400 错误（"Registration failed"），根本不会创建账号。

实测被封的域名包括：
- `wshu.net`（Mail.tm 的常用域名）
- `guerrillamailblock.com`（Guerrilla Mail，域名本身就包含"guerrilla"和"mail"，太容易被识别）

未被封的：
- `6r.dogmrp.com`、`pi.blaizesmp.net`、`ze.for4u.net` 等（tempmail.lol 使用的动态域名，看起来像正常网站）

### 防线 3：频率限制 — "同一个人别注册太多"

网站会记住你的网络地址（IP），如果同一个 IP 短时间内注册太多次，就会拒绝：

- 大约每小时每个 IP 只允许 5 次注册
- 第一次超限时返回"注册失败，请联系客服"（故意伪装成普通错误）
- 继续尝试则直接返回"请求太多"

---

## 四、我们怎么突破这四道防线

### 对付验证码 → Capsolver（第三方解码服务）

我们不需要自己去"看图识字"。有专门的服务商（Capsolver）提供 API：

```
我们 → 告诉 Capsolver："帮我解这个网站的验证码"
Capsolver → 在他们的服务器上解决 → 返回一个"通行证"字符串
我们 → 把这个通行证提交给 Serper
```

就像你不想自己排队取号，花钱请了个跑腿帮你排队拿号。

### 对付邮箱验证 + 域名黑名单 → 临时邮箱服务（三家自动切换）

我们用免费的一次性邮箱服务，它们提供 API 接口（程序可以直接调用）：

```
我们 → 调用临时邮箱服务创建一个邮箱（比如 abc123@pi.blaizesmp.net）
注册时填这个邮箱
Serper → 发验证邮件到这个邮箱
我们 → 通过 API 读取邮件 → 提取验证链接
我们 → 调用验证 API → 账号激活
```

就像去便利店借了个一次性手机号来收短信验证码。

**关键发现：不是所有临时邮箱都能用。** Serper 有一次性邮箱域名黑名单，像 `wshu.net`（Mail.tm）和 `guerrillamailblock.com`（Guerrilla Mail）已经在名单上。所以我们优先使用 **tempmail.lol**，因为它的域名（如 `6r.dogmrp.com`）看起来不像一次性邮箱，能通过黑名单检查。

三家服务作为 fallback 链，但优先级已调整：

```
尝试 tempmail.lol  → 成功就用（域名不在黑名单上，优先使用）
  ↓ 失败
尝试 Mail.tm       → 成功就用（域名可能被封，作为备选）
  ↓ 失败
尝试 Guerrilla Mail → 成功就用（域名大概率被封，最后兜底）
  ↓ 失败
报错退出
```

任何一家挂了，自动用下一家，不需要手动改代码。

### 对付频率限制 → IPRoyal 旋转住宅代理（已实现）

每次注册用不同的网络地址，让网站以为是不同的人在注册。详见第七节。

## 五、完整的自动注册流程

把上面三个方案组合起来，一次完整注册是这样的：

```
第 1 步（同时进行，节省时间）:
  ├── 创建临时邮箱          → 调用 tempmail.lol（优先，域名不在黑名单上）
  ├── 解 reCAPTCHA 验证码    → 调用 Capsolver
  └── 解 Turnstile 验证码    → 调用 Capsolver

第 2 步: 提交注册请求（通过 IPRoyal 住宅代理）
  → 把邮箱、密码、两个验证码通行证一起发给 Serper
  → 如果被频率限制，换 IP + 等待后重试（最多 5 个 IP）

第 3 步: 登录
  → 再解一个 Turnstile 验证码
  → 用邮箱密码登录，拿到登录凭证

第 4 步: 收验证邮件
  → 每隔几秒去 tempmail.lol 查一次有没有新邮件
  → 找到 Serper 发的邮件 → 解码 QP 编码 → 提取验证链接

第 5 步: 激活账号
  → 从验证链接中提取 token
  → 调用 POST /users/verify-email API 激活（不需要浏览器）

第 6 步: 拿钥匙
  → 重新登录 → 调用 API 获取 API Key
  → 保存到本地文件
```

一次注册大约 15-20 秒（主要时间花在等验证码解决和等邮件）。

## 六、两个关键的架构决策

### 决策 1：不用浏览器，直接发 HTTP 请求

最初的版本会启动一个完整的 Chrome 浏览器（虽然你看不到窗口），像真人一样在页面上操作。

后来发现：Serper 的所有操作其实都可以通过直接发送网络请求完成，根本不需要浏览器。

**类比**：你想从餐厅点外卖。
- 旧方案：开车去餐厅、坐下来、看菜单、跟服务员点菜、等上菜、打包带走
- 新方案：直接打电话下单

效果一样，但新方案快得多，也不需要一辆车。

**好处**：
- 启动时间从 3-5 秒降到 0
- 内存占用从 200MB 降到 10MB
- 不需要安装 Chrome 浏览器
- 代码更简单，更不容易出错

### 决策 2：第 1 步三件事并发执行

创建邮箱和解两个验证码之间没有依赖关系，可以同时做。

**类比**：你要准备三样东西——买菜、洗衣服、充电。
- 串行：买菜 → 洗衣服 → 充电（总共 30 分钟）
- 并行：同时进行（只需要最久那个的时间，比如 12 分钟）

## 七、IP 频率限制与解决方案（已实现）

### 问题

同一个 IP 每小时只能注册约 5 次。超限后服务器的行为很"狡猾"：

- 第一次超限：返回 **"注册失败，请联系客服"**（故意伪装成普通错误，不告诉你是频率限制）
- 继续尝试：返回 **"请求太多"**（这才暴露真实原因）

如果用固定 IP，注册几次就会被拦住。

### 解决方案：IPRoyal 旋转住宅代理

我们使用 IPRoyal 的旋转代理服务。每次注册自动分配一个不同的住宅 IP 地址：

```
注册账号 A → 代理分配 IP-1（美国某家庭宽带） → 注册成功
注册账号 B → 代理分配 IP-2（德国某家庭宽带） → 注册成功
注册账号 C → 代理分配 IP-3（日本某家庭宽带） → 注册成功
```

**类比**：你想给同一家店投多张优惠券，但店员认脸。解决办法是每次让不同的朋友替你去投。IPRoyal 在全球有 200 万个"朋友"（住宅 IP），用不完。

**为什么选住宅代理而不是机房代理？** 因为住宅 IP 来自真实的家庭宽带（比如某个美国人家里的网络），网站很难区分它和真人。机房 IP 一看就是服务器，容易被封。

### 两个关键的技术细节

#### 细节 1：粘性会话 — 同一账号内 IP 不变

一次注册流程有多个步骤都要访问 Serper（注册、登录、验证、取钥匙）。如果每个请求都换 IP，服务器会发现"刚刚登录的人突然从另一个城市出现了"，然后作废登录状态。

所以我们用**粘性会话（Sticky Session）**：给每次注册生成一个唯一的会话 ID，嵌入代理的密码中。IPRoyal 看到同一个会话 ID，就会一直分配同一个 IP：

```
注册账号 A：会话 ID = "abc12345" → 全程走 IP-1（5 分钟内不变）
注册账号 B：会话 ID = "xyz67890" → 全程走 IP-2（5 分钟内不变）
```

不同账号 = 不同会话 ID = 不同 IP。同一账号内所有请求 = 同一个 IP。

#### 细节 2：只有 Serper 的请求走代理

整个流程中访问了三个不同的服务。只有 Serper 被频率限制，所以只代理它的请求：

| 服务 | 作用 | 走代理吗 | 原因 |
|------|------|----------|------|
| Serper.dev | 注册/登录/取钥匙 | **走代理** | 被频率限制，需要换 IP |
| 临时邮箱 | 创建邮箱/收邮件 | 直连 | 没有限制 |
| Capsolver | 解验证码 | 直连 | 没有限制 |

这样既省代理流量（省钱），又让非 Serper 请求更快。

### 代理质量保障：预检机制

住宅代理的 IP 来自真实用户的网络，偶尔会遇到质量差的 IP（网络慢、连不上目标网站）。我们在注册前会先做一次"预检"：

```
拿到代理 IP → 试着访问 Serper → 能连通吗？
  能连通 → 用这个 IP 注册
  连不通 → 自动换一个 IP → 再试（最多试 5 次）
```

这样确保每次注册都用一个健康的代理 IP。

## 八、项目文件一览

```
serper注册机/
├── config.py           配置文件（网址、密钥、代理设置等）
├── .env                敏感配置（API 密钥、代理凭证，不上传到网上）
├── email_module.py     临时邮箱模块（tempmail.lol → Mail.tm → Guerrilla，自动切换）
├── captcha_module.py   验证码解决模块（Capsolver）
├── proxy_module.py     旋转代理模块（IPRoyal 粘性会话）
├── register.py         单次注册的完整流程
├── batch_register.py   批量注册（控制同时运行几个）
├── requirements.txt    需要安装的软件包列表
└── api_keys.json       输出结果（注册成功的账号和钥匙）
```

每个模块各管一件事，register.py 把它们串起来。

## 十、踩过的坑

### 坑 1：一次性邮箱域名被封

最初优先使用 Mail.tm（域名 `wshu.net`），前 3 个账号注册成功。之后所有注册都返回 400 错误，换了 10 个不同的住宅 IP 也没用。

排查过程：同时测试三个邮箱服务注册——Mail.tm 失败、Guerrilla Mail 失败、tempmail.lol **成功**。确认是邮箱域名在 Serper 的黑名单上。

修复：把 tempmail.lol 调整为第一优先级，因为它每次创建邮箱用不同的域名（像 `6r.dogmrp.com`），不在常见黑名单中。

### 坑 2：邮件编码导致验证 token 损坏

tempmail.lol 返回的邮件正文使用了 quoted-printable 编码。验证链接中的 `token=abc123` 在邮件中变成了 `token=3Dabc123`（`=3D` 是 `=` 号的 QP 编码）。直接提取的 token 多了 `3D` 前缀，导致验证 API 返回 404。

修复：在提取链接前先用 `quopri` 模块解码邮件正文。

### 坑 3：验证链接是前端页面，不是真正的 API

Serper 发的验证邮件链接指向 `https://serper.dev/confirm-email?token=xxx`，这是一个 Next.js 前端页面，需要浏览器执行 JavaScript 才能完成验证。纯 HTTP 请求访问这个链接只会拿到 HTML 模板。

解决：阅读前端 JS 源码，发现它实际上调用了 `POST /users/verify-email` API。直接调用这个 API 传入 token 即可完成验证，无需浏览器。
